## Recommended execution order

1. **001, 002, 003, 004** (the four HIGH-severity plans) — independent of each other, safe to execute in parallel/any order. These are the highest-leverage fixes: two close real accessibility gaps (reduced-motion), one fixes a genuinely janky gesture interaction, one removes a real paint-cost problem from a site-wide layout wrapper.
2. **005** — touches `components/Projects.tsx`'s `x` line and `components/Hero.tsx`'s `y`/scroll-down lines. If 002 already landed, `Projects.tsx`'s `useTransform` call will already read `reduced ? 0 : -overflow` — plan 005's excerpt accounts for this (see its Target section's inline comment). Apply after 001/002 to avoid re-deriving the same line twice; not a hard blocker if applied first, just re-read the current file state per each plan's own "if drift, STOP" rule.
3. **006** — touches the same transition-string line in `BlogPostsCarousel3D.tsx` that plan 003 modifies (the `settleMs`-based template literal). Apply after 003 so 006 wraps the *already-dynamic* duration in the token, rather than 006 landing first and 003 having to re-discover where the curve is. Not a hard blocker either way — both plans include the current-state excerpt they expect and will STOP on drift rather than silently conflicting.
4. **007** — independent, but if you're doing a single pass over `KuumaCompanion.tsx`, note that plan 001 also edits this file (a different section — the mouse-follow effect and the `<style>` block, not the bubble/panel JSX) and plan 008 also edits this file (the `showBubble` function and its three call sites, again a different section). All three plans include enough surrounding context in their excerpts to apply independently and in any order, but applying them one at a time (not by hand-merging diffs) avoids confusion.
5. **008** — independent; see note above.
6. **009** — fully independent, smallest plan, good candidate to execute first if you want a quick low-risk win before tackling the larger ones.
7. **010** — Part A is fully independent (`ExplorationBadge.tsx`). Part B touches `KuumaCompanion.tsx`'s message-list render and imports `AnimatePresence`/`motion` — if plan 007 already added that import to the same file, part B should reuse it rather than adding a duplicate import line (both plans call this out explicitly).

**No plan is a hard blocker for another.** The "depends on" column above is about avoiding redundant/conflicting edits to the same lines within one file, not about correctness — every plan documents the current-code excerpt it expects and instructs the executor to STOP and report rather than improvise if that excerpt has already drifted (e.g. because a dependency plan landed first and changed the surrounding lines).
